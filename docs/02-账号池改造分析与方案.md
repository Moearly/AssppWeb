# AssppWeb 账号池共享架构改造方案

> 解决当前架构无法支持账号池共享的问题

---

## 🔴 当前架构的核心问题

### 问题 1：账号存储在浏览器本地

**当前实现**：

```typescript
// frontend/src/store/accounts.ts
const DB_NAME = "asspp-accounts";  // 浏览器 IndexedDB
const STORE_NAME = "accounts";

// 账号存储在每个用户的浏览器中
await db.put(STORE_NAME, {
  email: 'user@icloud.com',
  password: 'secret',
  passwordToken: 'token',
  cookies: {...}
});
```

**问题**：
- ❌ 每个用户浏览器有自己的账号副本
- ❌ 无法在多个用户之间共享账号池
- ❌ 管理员添加账号后，普通用户看不到
- ❌ 账号健康状态无法集中监控

---

### 问题 2：零信任架构与账号池共享冲突

**当前架构**：

```
浏览器（IndexedDB）
    ↓ 账号凭据仅存在于此
    ↓ 服务器永远看不到
libcurl.js (WASM)
    ↓ TLS 1.3 加密
Wisp 盲代理
    ↓ 加密流量
Apple 服务器
```

**核心矛盾**：

| 需求 | 零信任架构 | 账号池共享 |
|------|----------|----------|
| 账号存储位置 | 浏览器本地 | 服务器端 |
| 服务器是否可见凭据 | ❌ 不可见 | ✅ 必须可见 |
| 多用户共享 | ❌ 不支持 | ✅ 必须支持 |

**结论**：**零信任架构** 和 **账号池共享** 是**互斥**的！

---

### 问题 3：Apple 协议必须在客户端执行

**当前流程**：

```
浏览器执行 Apple 协议:
1. libcurl.js 建立 TLS 1.3 连接
2. 发送认证请求（包含密码）
3. 发送购买请求（包含 passwordToken）
4. 获取下载 URL 和 SINF

服务器仅负责:
- Wisp 盲代理（TCP 转发）
- 下载 IPA 文件
- 注入 SINF
```

**如果改为账号池共享**：

```
服务器必须执行 Apple 协议:
1. 服务器存储账号凭据 ⚠️
2. 服务器调用 Apple API ⚠️
3. 服务器管理 passwordToken 和 cookies ⚠️

问题：
- ❌ 服务器能看到所有账号的密码
- ❌ 服务器成为攻击目标
- ❌ 违背零信任原则
```

---

## ✅ 解决方案：双模式架构

### 方案概述

AssppWeb 需要支持 **两种模式**，让用户根据场景选择：

| 模式 | 适用场景 | 账号存储 | 安全性 | 共享能力 |
|------|---------|---------|--------|---------|
| **模式 1：零信任模式** | 个人用户 | 浏览器 IndexedDB | ⭐⭐⭐⭐⭐ | ❌ 不支持共享 |
| **模式 2：账号池模式** | 团队/机构 | 服务器数据库 | ⭐⭐⭐ | ✅ 支持共享 |

---

## 🏗️ 新架构设计

### 模式 1：零信任模式（保持现有实现）

```
┌─────────────────────────────────────────────┐
│  用户 A 浏览器                               │
│  ├─ IndexedDB: [账号 A]                     │
│  ├─ libcurl.js: TLS 1.3 加密                │
│  └─ Apple 协议执行                           │
└──────────────┬──────────────────────────────┘
               ↓ Wisp 盲代理
┌──────────────┴──────────────────────────────┐
│  服务器                                      │
│  ├─ Wisp 中继（看不到凭据）                  │
│  └─ IPA 下载 + SINF 注入                     │
└─────────────────────────────────────────────┘
```

**特点**：
- ✅ 服务器看不到凭据
- ✅ 最高安全性
- ❌ 每个用户管理自己的账号
- ❌ 无法共享

---

### 模式 2：账号池模式（新增）

```
┌─────────────────────────────────────────────┐
│  管理员浏览器                                │
│  └─ 管理界面：添加/删除账号池               │
└──────────────┬──────────────────────────────┘
               ↓ HTTPS API (加密传输)
┌──────────────┴──────────────────────────────┐
│  服务器                                      │
│  ├─ SQLite 数据库                            │
│  │  └─ 账号池表 (AES-256-GCM 加密)          │
│  ├─ Apple 协议执行（服务器端）               │
│  │  ├─ 认证                                 │
│  │  ├─ 购买                                 │
│  │  └─ 下载                                 │
│  └─ 智能账号选择                             │
│     ├─ 负载均衡                              │
│     ├─ 健康检查                              │
│     └─ 频率限制                              │
└──────────────┬──────────────────────────────┘
               ↓
┌──────────────┴──────────────────────────────┐
│  普通用户浏览器                              │
│  └─ 简化界面：选择应用 → 一键下载           │
└─────────────────────────────────────────────┘
```

**特点**：
- ✅ 账号集中管理
- ✅ 多用户共享
- ✅ 自动负载均衡
- ⚠️ 服务器能看到凭据（加密存储）
- ⚠️ 需要信任服务器管理员

### 第一阶段：数据库设计

#### 1. 账号池表

```sql
CREATE TABLE account_pool (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  email TEXT NOT NULL UNIQUE,
  email_hash TEXT NOT NULL,           -- SHA-256(email)
  encrypted_data TEXT NOT NULL,       -- AES-256-GCM 加密的凭据
  country TEXT NOT NULL,
  device_identifier TEXT NOT NULL,
  pod INTEGER,
  status TEXT DEFAULT 'active',       -- active | disabled | expired
  last_used_at DATETIME,
  usage_count INTEGER DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  updated_at DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_status ON account_pool(status);
CREATE INDEX idx_country ON account_pool(country);
```

**encrypted_data 内容**（加密前）：

```json
{
  "password": "账号密码",
  "passwordToken": "Apple Token",
  "cookies": { "..." },
  "DSID": "..."
}
```

**加密方法**：

```typescript
// 使用服务器端密钥加密
const ENCRYPTION_KEY = process.env.ACCOUNT_POOL_KEY; // 256-bit 密钥

async function encryptAccountData(data: AccountCredentials): Promise<string> {
  const key = await crypto.subtle.importKey(
    'raw',
    Buffer.from(ENCRYPTION_KEY, 'hex'),
    { name: 'AES-GCM', length: 256 },
    false,
    ['encrypt']
  );
  
  const iv = crypto.getRandomValues(new Uint8Array(12));
  const encrypted = await crypto.subtle.encrypt(
    { name: 'AES-GCM', iv },
    key,
    new TextEncoder().encode(JSON.stringify(data))
  );
  
  return JSON.stringify({
    iv: Buffer.from(iv).toString('base64'),
    data: Buffer.from(encrypted).toString('base64')
  });
}
```

#### 智能账号选择器

```typescript
// backend/src/services/accountSelector.ts

export class AccountSelector {
  /**
   * 为下载任务选择最佳账号
   */
  async selectAccount(country: string): Promise<number> {
    // 1. 筛选：状态正常 + 地区匹配
    const candidates = await db.all(`
      SELECT id, usage_count, last_used_at
      FROM account_pool
      WHERE status = 'active'
        AND country = ?
      ORDER BY usage_count ASC, last_used_at ASC
      LIMIT 5
    `, [country]);
    
    if (candidates.length === 0) {
      throw new Error(`No available accounts for country: ${country}`);
    }
    
    // 2. 负载均衡：选择使用次数最少的
    const selected = candidates[0];
    
    // 3. 频率检查：确保冷却时间
    const lastUsed = new Date(selected.last_used_at).getTime();
    const cooldown = 5 * 60 * 1000; // 5分钟冷却
    
    if (Date.now() - lastUsed < cooldown) {
      // 选择下一个账号
      if (candidates.length > 1) {
        return candidates[1].id;
      }
      // 所有账号都在冷却，等待
      throw new Error('All accounts are cooling down');
    }
    
    // 4. 更新使用统计
    await db.run(`
      UPDATE account_pool
      SET usage_count = usage_count + 1,
          last_used_at = CURRENT_TIMESTAMP
      WHERE id = ?
    `, [selected.id]);
    
    return selected.id;
  }
}
```

#### 健康检查服务

```typescript
// backend/src/services/healthCheck.ts

export class HealthCheckService {
  /**
   * 检查单个账号健康状态
   */
  async checkAccount(accountId: number): Promise<HealthStatus> {
    try {
      const client = new AppleClient(accountId);
      
      // 尝试认证
      await client.authenticate();
      
      // 记录健康状态
      await db.run(`
        INSERT INTO health_log (account_id, status)
        VALUES (?, 'healthy')
      `, [accountId]);
      
      return { status: 'healthy' };
      
    } catch (error) {
      let status = 'error';
      
      if (error.failureType === 2034 || error.failureType === 2042) {
        status = 'token_expired';
      }
      
      // 记录错误
      await db.run(`
        INSERT INTO health_log (account_id, status, error_message)
        VALUES (?, ?, ?)
      `, [accountId, status, error.message]);
      
      // 如果 token 过期，标记账号为需要重新认证
      if (status === 'token_expired') {
        await db.run(`
          UPDATE account_pool
          SET status = 'expired'
          WHERE id = ?
        `, [accountId]);
      }
      
      return { status, message: error.message };
    }
  }
  
  /**
   * 定时检查所有账号
   */
  async runScheduledCheck(): Promise<void> {
    const accounts = await db.all(`
      SELECT id FROM account_pool WHERE status = 'active'
    `);
    
    for (const account of accounts) {
      await this.checkAccount(account.id);
      // 避免频繁请求，间隔10秒
      await sleep(10000);
    }
  }
}

// 启动定时任务（每6小时）
setInterval(() => {
  new HealthCheckService().runScheduledCheck();
}, 6 * 60 * 60 * 1000);
```

## 📊 两种模式对比

| 维度 | 零信任模式 | 账号池模式 |
|------|----------|----------|
| **安全性** | ⭐⭐⭐⭐⭐ 服务器看不到凭据 | ⭐⭐⭐ 服务器可见凭据（加密） |
| **账号管理** | 每个用户自己管理 | 管理员集中管理 |
| **共享能力** | ❌ 不支持 | ✅ 多用户共享 |
| **负载均衡** | ❌ 手动切换 | ✅ 自动选择 |
| **健康监控** | ❌ 用户自行处理 | ✅ 自动检查 |
| **用户体验** | 需要添加账号 | 一键下载 |
| **适用场景** | 个人使用 | 团队/机构 |
| **部署复杂度** | 简单 | 中等 |

## 总结

你说得对，**当前的零信任架构无法支持账号池共享**。必须做出选择：

1. **保留零信任** = 放弃共享
2. **实现共享** = 放弃零信任

**推荐方案**：**双模式并存**

- 个人用户继续使用零信任模式（最安全）
- 团队用户使用账号池模式（更方便）

这样既满足你的需求，又不会完全牺牲安全性。
